🧪 Salon Manager 통합 테스트 체크리스트 (Comprehensive Test Plan)
현재까지 구현된 모든 기능에 대한 디버깅 및 테스트 항목입니다. 자동화된 테스트 코드가 없으므로, 수동 테스트(Manual Testing) 또는 **API 테스트 도구(Postman/Insomnia)**를 통해 검증해야 합니다.

1. 인증 및 세션 (Authentication & Session)
로그인 (Login)
 올바른 이메일/비밀번호로 로그인 성공 여부 (토큰 발급)
 틀린 계정 정보로 로그인 시도 시 에러 메시지 노출 (401 Unauthorized)
 [중요] 새로고침 시 로그인 상태 유지 (Refresh Token 작동 확인)
로그아웃 (Logout)
 로그아웃 버튼 클릭 시 쿠키 삭제 및 로그인 페이지 리다이렉트
 로그아웃 후 뒤로가기 시 보안 페이지 진입 불가 확인
2. 일정 및 캘린더 (Schedule & Calendar)
조회 (View)
 [중요] 월간(Month), 주간(Week), 일간(Day) 뷰 전환 정상 동작
 디자이너별 스케줄 필터링 동작 (선택한 디자이너의 예약만 표시)
 영업 시간(오픈/마감) 외 시간대 렌더링 확인 (Start/End Time 설정)
상호작용 (Interaction)
 빈 시간 클릭 시 '예약 생성 모달' 오픈 (날짜/시간 자동 입력 확인)
 예약 블록 클릭 시 '예약 상세 모달' 오픈
3. 예약 관리 (Reservations)
예약 생성 (Create)
 신규 고객 / 기존 고객 검색 후 예약 등록
 필수 값(메뉴, 디자이너, 시간) 누락 시 유효성 검사
 [버그 체크] 영업 시간 외 예약 시도 시 경고 또는 차단 여부
 [버그 체크] 중복된 시간대(Overlapping) 예약 시도시 처리 로직 확인
예약 상태 변경 (Update Status)
 상태 변경: 대기(Pending) -> 확정(Confirmed) -> 완료(Completed) -> 취소(Canceled) -> 노쇼(NoShow)
 상태 변경 시 캘린더 내 색상 동기화 확인
예약 수정/삭제 (Edit/Delete)
 예약 시간, 메뉴 변경 후 저장 시 반영 여부
 예약 삭제(Soft Delete) 후 캘린더에서 사라지는지 확인
4. 미용실 설정 (Shop Settings)
운영 설정
 영업 시간(Open/Close Time) 변경 후 캘린더 렌더링 즉시 반영 여부
 휴무일 설정 기능 (아직 구현 안됨? 확인 필요)
디자이너 관리 (Designer Management)
 디자이너 추가 (신규 계정 생성 포함) 정상 동작
 디자이너 정보 수정 (근무 시간, 소개글 등)
 디자이너 퇴사(비활성화) 처리 시 예약 생성 불가 확인
메뉴 관리 (Menu Settings) - [최신 변경 사항]
 [NEW] '메뉴 카테고리' 탭에서 카테고리 추가/순서변경
 [NEW] '시술 메뉴' 탭에서 카테고리 선택하여 메뉴 추가
 메뉴 가격/소요시간 수정 반영 확인
5. 매출 관리 (Sales Dashboard)
집계 정확성
 일별/월별 매출 총액이 PAYMENTS 테이블 데이터와 일치하는지 확인
 '메뉴별 매출' 탭에서 카운트 및 금액 합계 정확성
 결제 수단(카드/현금)별 비율 그래프 표시 확인
기간 조회
 날짜 네비게이션(이전 달/다음 달) 동작 확인
6. 고객 관리 (Customer Management)
고객 목록
 이름/전화번호 검색 기능 동작
 고객 등급(VIP, NEW, CAUTION) 라벨 표시 정확성 확인
고객 상세
 방문 히스토리(이전 예약 기록) 불러오기 확인
 관리자 메모(Customer Memo) 작성 및 저장 확인
🛠 중점 디버깅 포인트 (Priority Debugging Areas)
최근 코드 변경이 많았던 부분을 우선 점검해야 합니다.

메뉴 카테고리 분리 영향도:
예약 생성 모달에서 메뉴 선택 시, 변경된 카테고리 구조(Category -> Menu)로 잘 뜨는지 확인해야 합니다.
기존 예약 데이터들의 메뉴 정보가 깨지지 않았는지 확인 (RESERVATION_ITEMS와 MENUS 조인).
DTO 유효성 검사:
백엔드에서 
CreateMenuDto
 등에 type 필드 추가 후, 실제 API 호출 시 400 에러가 완전히 사라졌는지 재확인.
데이터베이스 정합성:
yarn prisma db push 또는 
complete_init.sql
 실행 후, FK(외래키) 관계가 유효한지 확인.
